# `echelon.config.ts` Schema (Public)

This document defines the **public** top-level configuration surface for Echelon products.
It is intended to replace direct usage of repo-owned `controlplane.bindings.json` for most
agent/product code, while keeping backwards compatibility for existing ACP adopters.

Ticket coverage:
- TGA-164 schema documentation + migration path

---

## 1. Goals

1. Provide a single place for project-specific keys (tenant/auth/database + optional packs).
2. Exclude internal-only installer keys from the public schema.
3. Make migration from `controlplane.bindings.json` a mechanical mapping.

---

## 2. Public Shape (TypeScript)

In code, you should treat this as the canonical shape exported from your own `echelon.config.ts`.

```ts
import type { Pack } from 'agentic-control-plane-kit';

export type EchelonConfig = {
  /**
   * Public bindings.
   *
   * Note: this is a *public subset* of the kernel `Bindings`.
   * Internal-only keys (installer-generated) are intentionally omitted.
   */
  bindings: {
    tenant: {
      table: string;
      id_column: string;
      get_tenant_fn: string;
      is_admin_fn: string;
    };
    auth: {
      keys_table: string;
      key_prefix: string;
      prefix_length: number;
      key_hash_column: string;
      key_prefix_column: string;
      scopes_column: string;
    };
    database: {
      adapter: 'supabase' | 'prisma' | 'drizzle' | 'custom';
      [key: string]: any;
    };

    /**
     * Optional: the kernel uses this for audit/context.
     * For most adopters this should be set once and remain stable.
     */
    integration?: string;
  };

  /**
   * Packs installed into your `/manage` router.
   * Domain packs are typically repo-owned; built-ins can be imported from `agentic-control-plane-kit/packs`.
   */
  packs: Pack[];
};
```

## 3. Example: Minimal `echelon.config.ts`

```ts
import { defineConfig } from 'agentic-control-plane-kit';
import { iamPack, webhooksPack, settingsPack } from 'agentic-control-plane-kit/packs';

const config = defineConfig({
  bindings: {
    tenant: {
      table: 'brands',
      id_column: 'id',
      get_tenant_fn: 'get_my_brand_id',
      is_admin_fn: 'is_platform_admin',
    },
    auth: {
      keys_table: 'api_keys',
      key_prefix: 'ock_',
      prefix_length: 12,
      key_hash_column: 'key_hash',
      key_prefix_column: 'prefix',
      scopes_column: 'scopes',
    },
    database: {
      adapter: 'supabase',
      connection_env: 'SUPABASE_SERVICE_ROLE_KEY',
    },
    integration: 'myapp',
  },
  packs: [iamPack, webhooksPack, settingsPack],
});

export default config;
```

## 4. Migration from `controlplane.bindings.json`

### What to copy

For existing adopters using `controlplane.bindings.json`, migrate by copying the *public* keys into:
`echelon.config.ts -> config.bindings`.

Mapping:

- `tenant` -> `bindings.tenant`
- `auth` -> `bindings.auth`
- `database` -> `bindings.database`
- `packs.enabled` (if present) -> map to actual Pack objects you install in `packs`
  - built-in packs: use `agentic-control-plane-kit/packs`
  - domain packs: import your repo-owned pack
- ignore installer/runtime-only keys:
  - keys like `kernelId`, `base_path`, `endpoint_path`, and any internal wiring keys
  - keep only the stable public subset

### Why this works

`controlplane.bindings.json` already stores the stable repo-specific truth.
The installer can still generate runtime wiring keys as needed; product code should not depend on them.

---

## 5. Internal-only keys excluded from public schema

Public schema should not require keys that are:
- generated by the installer,
- environment-specific,
- used for runtime wiring/registration rather than product semantics.

If you find a key in your current `controlplane.bindings.json` that feels like one of the above,
prefer to exclude it from `echelon.config.ts` and let the installer/runtime layer provide it.

---

## 6. Compatibility guarantees for current ACP adopters

This public config model is intended to be **additive**, not a flag day rewrite.

Current adopters such as existing `/manage` deployments should remain valid if they continue to use:

- generated `controlplane.bindings.json`
- existing installer output
- current `createManageRouter(...)` embedding flow

Recommended migration path:

1. Keep the existing `controlplane.bindings.json` as the runtime source of truth.
2. Add an `echelon.config.ts` facade for product-facing code.
3. Use SDK helpers to bridge the two shapes:
   - `fromBindings(bindings, packs)` to expose a public config from a legacy bindings file
   - `toBindings(config)` to feed the current kernel router from the public config
4. Migrate public docs and app code first; migrate installer/runtime internals later.

Non-goal for this phase:

- forcing existing ACP adopters to stop using `controlplane.bindings.json`
- changing the current `/manage` runtime contract
- changing existing Repo B / Repo C integration expectations for already-deployed SaaS apps
